JVM

一. 内存与垃圾回收

1. JVM与Java体系结构

Java是跨平台的语言，JVM是跨语言的平台

虚拟机（Virtual Machine）可以分为：

1）系统虚拟机：对物理计算机的仿真，VMware

2）程序虚拟机：专门执行单个计算机程序，Java虚拟机

JVM运行在操作系统上，与硬件没有直接交互

Java编译器输入的指令流基本是基于栈的指令集架构：

  1. 设计和实现简单，适用于资源受限的系统
  2. 使用零地址指令，执行过程依赖于操作栈，指令集更小（8位<16位）
  3. 不需要硬件支持，可移植性好

基于寄存器架构特点：

  1. 指令集架构完全依赖硬件，可移植性差
  2. 完成操作需要的指令更少，性能优秀，执行更高效
  3. 通常以一地址指令、二地址指令、三地址指令为主
JVM生命周期

虚拟机的启动：Java虚拟机的启动是通过引导类加载器（bootstrap class loader）创建一个初始类（initial class）来完成的，这个类是由虚拟机的具体实现指定的

虚拟机的执行：程序开始执行时JVM才运行，程序结束时JVM停止。执行Java程序实际上在执行一个叫做Java虚拟机的进程

虚拟机的退出：程序正常执行结束；程序执行中遇到异常或错误而异常终止；操作系统错误导致Java虚拟机进程终止；某线程调用Runtime类或System类的exit方法，或Runtime类的halt方法，且Java安全管理器允许此次操作；JNI（Java Native Interface）规范描述了用JNI Invocation API来加载或卸载Java虚拟机时的退出情况

JVM发展历程

  • Sun Classic VM

    世界上第一款商用Java虚拟机，JDK1.4时完全淘汰

    只提供解释器，使用JIT编译器需要外挂，解释器和编译器不能配合工作

    HotSpot内置了此虚拟机

  • Exact VM

    JDK1.2时发布，具备现代高性能虚拟机的雏形：热点探测、编译器与解释器混合工作模式

    Exact Memory Management：准确式内存管理，虚拟机可以知道内存中某个位置的数据具体是什么类型

    只在Solaris平台短暂使用

  • HotSpot VM

    Sun/Oracle JDK和Open JDK的默认虚拟机

    HotSpot就是指热点代码探测技术：通过计数器找到最具编译价值代码，触发即时编译或栈上编译；编译器与解释器协同工作

  • BEA JRockit

    专注于服务端应用，不关注程序启动速度，JRockit不包含解释器

    JRockit JVM是最快的JVM

  • IBM J9

    IBM Technology for Java Virtual Machine，简称IT4J，内部代号J9

    定位与HotSpot接近，应用于IBM的各种Java产品

2. 类加载子系统

类加载子系统包括加载阶段、链接阶段、初始化阶段

类加载子系统负责从文件系统或网络中加载class文件，class文件开头有特定标识

ClassLoader只负责class文件的加载，由ExecutionEngine判断是否可以运行

加载的类信息存放于方法区，方法区还会存放运行时常量池信息

  • 加载（Loading）
  1. 通过一个类的全限定名获取此类的二进制字节流
  2. 将该字节流代表的静态存储结构转化为方法区的运行时数据结构
  3. 在内存中生成代表这个类的java.lang.Class对象，作为方法区这个类各种数据的访问入口
  • 链接（Linking）

  1. 验证（Verify）：

    确保class文件的字节流中包含信息符合当前虚拟机要求，保证被加载类的正确性，不会危害虚拟机自身安全

    包括四种验证：文件格式验证、元数据验证、字节码验证、符号引用验证

  2. 准备（Prepare）：

    为类变量分配内存并设置该类变量的默认初始值（零值）

    final修饰的static在编译时就已被分配值，准备阶段会显式初始化

    不会初始化实例变量，类变量在方法区，实例变量随对象一起分配到Java堆中

  3. 解析（Resolve）：

    将常量池内的符号引用转换为直接引用的过程

    符号引用是用一组符号描述所引用的目标；直接引用就是直接指向目标的指针、相对偏移量或一个间接定位到目标的句柄

    解析主要针对类或接口、字段、类方法、接口方法、方法类型等，对应常量池中的CONSTANT_Class_info、CONSTANT_Fieldref_info、CONSTANT_Methodref_info等

  • 初始化（Initialization）

    初始化阶段就是执行类构造器方法<clinit>()的过程

    此方法不需要定义，是javac编译器自动收集类中所有类变量的赋值动作和静态代码块中的语句合并而来

    <clinit>()不同于类的构造器，构造器是<init>()

    父类的<clinit>()先执行，子类的<clinit>()后执行

    虚拟机保证一个类的<clinit>()方法在多线程下被同步加锁

类加载器分类
  1. 虚拟机自带的加载器

  • 启动类加载器（引导类加载器，Bootstrap ClassLoader）

    使用C/C++语言实现（Java程序无法获取到其对象），嵌套在JVM内部

    用于加载Java核心库，提供JVM自身需要的类

    不继承自java.lang.ClassLoader，没有父加载器

    加载扩展类加载器、应用类加载器，为他们的父类加载器

    Bootstrap启动类加载器只加载包名为java、javax、sun等开头的类

  • 扩展类加载器（Extension ClassLoader）

    由Java语言编写

    派生于ClassLoader类，父类加载器为启动类加载器

    从java.ext.dirs系统属性指定的目录下加载类库，或从JDK安装目录的jre/lib/ext子目录下加载类库。此目录下用户创建的JAR也会由扩展类加载器加载

  • 应用程序类加载器（系统类加载器，AppClassLoader）

    Java语言编写

    派生于ClassLoader类，父类加载器为扩展类加载器

    负责加载环境变量classpath或系统属性java.class.path指定路径下的类库

    程序默认的类加载器，一般Java应用的类都由它完成加载

  1. 自定义类加载器（User-Defined ClassLoader）：所有派生于抽象类ClassLoader的类加载器

    隔离加载类、修改类加载的方式、扩展加载源、防止源码泄漏

双亲委派机制

Java虚拟机对class文件采用按需加载方式，需要使用到该类时才会将它的class文件加载到内存生成class对象。

加载class文件时采用双亲委派模式，即把请求交由父类处理，它是一种任务委派模式

工作原理：

  1. 如果一个类加载器收到了类加载的请求，它并不会自己先去加载，而是把这个请求委托给父类加载器去执行
  2. 如果父类加载器还存在其父类加载器，则进一步向上委托，依次递归，请求最终到达顶层的启动类加载器（自定义类加载器 -> 系统类加载器 -> 扩展类加载器 -> 引导类加载器）
  3. 如果父类加载器可以完成类加载任务，就成功返回；若父类加载器无法完成加载任务，子类加载器才会尝试自己去加载，这就是双亲委派模式

优点：避免类的重复加载，防止核心API被篡改

在JVM中表示两个class对象是否是同一个类存在两个必要条件：

  1. 类的完整类名必须一致（包括包名）
  2. 加载这个类的ClassLoader（ClassLoader实例对象）必须相同

JVM必须知道一个类型是由启动类加载器加载的还是由用户加载器加载的。如果一个类是由用户类加载器加载的，JVM会将这个类的一个引用作为类型信息的一部分保存在方法区。当解析一个类型到另一个类型的引用时，JVM需要保证两个类加载器是相同的

Java程序对类的使用方式分为：主动使用和被动使用（类不会初始化）

3. 运行时数据区

每个线程：独立包含程序计数器、栈、本地栈

线程间共享：堆、堆外内存（永久代或元空间、代码缓存）

线程

线程是程序的运行单元，JVM允许一个应用多线程并行执行

HotSpot JVM中，每个线程都与操作系统的本地线程直接映射。当一个Java线程准备好执行后，一个操作系统的本地线程也同时创建；Java线程执行终止后，本地线程也会回收

操作系统负责将线程安排调度到任何一个可用的CPU上，本地线程初始化成功就会调用Java线程的run()方法

HotSpot JVM后台线程包括：虚拟机线程、周期任务线程、GC线程、编译线程、信号调度线程

程序计数寄存器

  • 程序计数寄存器（Program Counter Register）：

    Register的命名源于CPU的寄存器，寄存器存储指令相关的现场信息，CPU只有把数据装载到寄存器才能运行

    JVM中的PC寄存器并非广义上的物理寄存器，是对物理PC寄存器的一种抽象模拟

    PC寄存器用来存储指向下一条指令的地址（将要执行的指令代码），由执行引擎读取下一条指令

    占用内存空间小，几乎可以忽略不计，是运行速度最快的存储区域

    每个线程都有自己的程序计数器，是线程私有的，生命周期与线程的生命周期一致

    线程在任何时间只能有一个方法在执行，称为当前方法。程序计数器会存储当前方法的JVM指令地址；如果在执行native方法，则是未指定值（undefined）

    程序计数器是控制流的指示器，分支、循环、跳转、异常处理、线程恢复等基础功能都依赖计数器完成

    是唯一一个在Java虚拟机规范中没有规定任何OutOfMemoryError情况的区域

  • 使用PC寄存器存储字节码指令的作用/为什么使用PC寄存器记录当前线程的执行地址？

    因为CPU需要在线程间不停切换，需要记录线程执行到哪里了。字节码解释器通过改变PC寄存器的值来明确下一条字节码指令

  • PC寄存器为什么会被设定为线程私有？

    多线程并发时，任意时刻一个处理器或多核处理器的一个内核只会执行某个线程中的一条指令，这样必然导致经常中断和恢复

    CPU在线程间不停切换，需要记录每个线程当前执行的位置，即每个线程分配一个PC寄存器

  • CPU时间片：

    CPU时间片即CPU分配给各个程序的时间，每个线程分配一段时间，称为它的时间片

    宏观上多个线程并行运行，微观上并发，引入时间片，程序轮流执行

虚拟机栈

由于跨平台性的设计，不同CPU架构不同，Java指令根据栈来设计，不能设计为基于寄存器的

栈是运行时的单位，堆是存储的单位

  • Java虚拟机栈（Java Virtual Machine Stack）：

    每个线程创建时都会创建一个虚拟机栈，内部保存一个个的栈帧（Stack Frame），对应着一次次的Java方法调用

    生命周期与线程一致

    主管Java程序的运行，保存方法的局部变量、部分结果，参与方法的调用和返回

    访问速度仅次于程序计数器

    方法开始执行，压栈；方法执行结束，弹栈

栈不存在垃圾回收问题，但可能有OOM

  • 栈可能出现的异常

    Java虚拟机规范允许Java栈的大小是动态的或固定不变

    StackOverflowError：使用固定大小虚拟机栈时，线程请求的栈容量超过Java虚拟机栈允许的最大容量

    OutOfMemoryError：虚拟机栈可以动态扩展，在尝试扩展时无法申请到足够内存，或创建新线程时没有足够的内存创建对应的虚拟机栈

IDEA中点击Run-Edit Configurations，在VM options中使用-Xss设置虚拟机栈空间：

  • 栈帧与方法一一对应：

    只有当前正在执行的方法的栈帧是有效的，该栈帧称为当前栈帧（Current Frame），对应的方法称为当前方法（Current Method），定义该方法的类称为当前类（Current Class）

  • 栈运行原理：

    不同线程间的栈帧不能相互引用

    如果当前方法调用了其他方法，被调用方法返回时会将执行结果给前一个栈帧，同时虚拟机丢弃当前栈帧，使前一个栈帧重新成为当前栈帧

    Java方法有两种返回方式：一种是正常退出，使用return指令；另一种是抛出异常（且方法中没有处理该异常）。两种方式都会导致栈帧被弹出

  • 栈帧的内部结构

    局部变量表（Local Variables）

    操作数栈（Operand Stack，或表达式栈）

    动态链接（Dynamic Linking，或指向运行时常量池的方法引用）

    方法返回地址（Return Address，或方法正常退出或异常退出的定义）

    一些附加信息

  • 局部变量表

    局部变量表也被称为局部变量数组或本地变量表

    定义为一个数字数组，主要用于存储基本数据类型、对象引用（reference），以及returnAddress类型

    局部变量表在栈中，是线程的私有数据，不存在数据安全问题

    局部变量表所需的容量大小是在编译器确定下来的，并保存在方法的Code属性的maximum local variables数据项中，运行期间不会改变局部变量表大小

    栈空间越大，方法调用次数就能更多；对于一个方法而言，参数和局部变量越多，局部变量表越大，栈帧就越大，调用该方法就需要更多栈空间，调用次数减少

    方法调用结束后，栈帧销毁，局部变量表也随之销毁

  • Slot

    局部变量表中基本存储单元是Slot（变量槽）

    参数值从局部变量数组的index0开始存放，到index-1位置结束

    局部变量表中，32位以内的类型占用一个slot（byte、short、char转换为int，boolean也转换为int，0表示false，非0表示true、returnAddress类型、float类型），64位类型占用两个slot（long、double）

    当一个实例方法被调用时，它的方法参数和局部变量会按照定义顺序复制到局部变量表的每一个slot上

    访问64位局部变量时，访问首个索引（如4和5表示一个slot，访问4）

    如果当前方法为构造方法或实例方法，则该对象引用this会存放在index为0的slot处，其余参数继续排列

  • 成员变量与局部变量

  1. 成员变量：使用前都经历过默认初始化赋值

    类变量：linking的prepare阶段：赋默认值 –> initialization阶段：显式赋值即静态代码块赋值

    实例变量：随着对象的创建，会在堆中分配空间，并进行默认赋值

  2. 局部变量：使用前必须进行显式赋值，否则编译报错

在栈帧中，局部变量表与性能调优关系最为密切。局部变量表中的变量也是重要的垃圾回收根节点，局部变量表中直接或间接引用的对象都不会被回收

  • 操作数栈（Operand Stack）

    每个栈帧中包含一个操作数栈，也称为表达式栈（Expression Stack）

    操作数栈在方法执行过程中，根据字节码指令将数据入栈或出栈，再将操作结果压入栈。例如复制、交换、求和等操作

    主要用于保存计算过程的中间结果，同时作为计算过程中变量临时的存储空间

    当一个方法开始执行时，创建新的栈帧，该方法的操作数栈是空的

    每个操作数栈都有明确的栈深度（使用数组实现），该深度在编译期就被确定下来，保存为Code属性的max_stack值

    栈中元素可以是任意Java数据类型：32bit类型占用一个单位深度，64bit类型占用两个单位深度

    如果被调用的方法有返回值，其返回值会被压入当前栈帧的操作数栈中

    Java虚拟机是基于栈的解释引擎，这里的栈指的是操作数栈

  • 栈顶缓存（Top-of-Stack Cashing）技术

    JVM基于栈结构使用零地址指令更加紧凑，但完成一项操作需要更多的入栈和出栈指令

    栈顶缓存技术将栈顶元素全部缓存到物理CPU的寄存器中，以此降低对内存的读写次数，提升执行引擎的执行效率

  • 动态链接（或指向运行时常量池的方法引用）

    每个栈帧都包含一个指向运行时常量池中该栈帧所属方法的引用，实现动态链接

    在编译时，所有变量和方法引用都作为符号引用（Symbolic Reference）保存在class文件的常量池中。例如描述一个方法调用其他方法时，就是通过常量池中指向方法的符号表示的，动态链接就是将这些符号引用转换为调用方法的直接引用

  • 方法的调用

    JVM中，将符号引用转换为调用方法的直接引用与方法的绑定机制有关

    静态链接：字节码文件被装载进JVM时，如果被调用的方法在编译期可知，且运行期保持不变，此时将调用方法的符号引用转换为直接引用的过程称为静态链接

    动态链接：被调用的方法在编译期无法被确定下来，只能在运行期将方法的符号引用转换为直接引用

    对应的方法绑定机制分别为早期绑定（Early Binding）、晚期绑定（Late Binding）。绑定是一个字段、方法或者类由符号引用被替换为直接引用的过程，只发生一次

    早期绑定：被调用的方法在编译期可知，且运行期保持不变，使用静态链接

    晚期绑定：被调用的方法在编译期无法被确定下来，只能在运行期根据实际类型绑定相关方法，使用动态链接

    Java中任何方法都具备虚函数特征，相当于C++语言的虚函数。如果不希望某个方法拥有虚函数特征，可以使用final关键字

  • 虚方法与非虚方法

    非虚方法：在编译期就确定了具体的调用版本，且运行时不变

    静态方法、私有方法、final方法、实例构造器、父类方法都是非虚方法

    其他方法称为虚方法

  • 虚拟机方法调用指令

    普通调用指令：

    invokestatic：调用静态方法，解析阶段确定唯一方法版本

    invokespecial：调用<init>方法、私有方法、父类方法，解析阶段确定唯一方法版本

    invokevirtual：调用所有虚方法

    invokeinterface：调用接口方法

    动态调用指令：

    invokedynamic：动态解析出需要调用的方法，然后执行

    普通调用指令固化在虚拟机内部，方法的调用执行不可人为干预，invokedynamic指令支持由用户确定方法版本。其中invokestatic指令和invokespecial指令调用的方法称为非虚方法，其余未加final修饰的方法为虚方法

    Java7增加invokedynamic指令，为了实现动态类型语言支持

  • 动态类型语言和静态类型语言

    静态类型语言在编译期进行类型检查，动态类型语言在运行期进行类型检查

    静态类型变量判断变量自身类型信息；动态类型语言判断变量值的类型信息，变量没有类型信息，变量值有类型信息

  • Java语言方法重写的本质

  1. 找到操作数栈顶的第一个元素所执行的对象的实际类型，记作C
  2. 如果在类型C中找到与常量中描述与简单名称都相符的方法，则进行权限校验，如果通过则返回该方法的直接引用，查找过程结束；如果不通过，返回java.lang.IllegalAccessError异常（属性或方法没有权限访问，一般会引起编译器异常。如果发生在运行时，说明一个类发生了不兼容的改变）
  3. 否则，按照继承关系从下向上依次对C的各个父类进行2的搜索和校验过程
  4. 如果始终没有找到合适的方法，抛出java.lang.AbstractMethodError异常

  • 虚方法表

    面向对象编程中，会频繁使用到动态分派。为提高性能，JVM在类的方法区建立一个虚方法表（Virtual method table），使用索引表代替查找

    每个类都有一个虚方法表，表中存放着各个方法的实际入口

    虚方法表在类加载的链接阶段创建并开始初始化，类的变量初始值准备完成后，虚方法表也初始化完毕

  • 方法返回地址（return address）

    存放调用该方法的pc寄存器的值

    方法正常退出时，调用者的pc寄存器值作为返回地址，即调用该方法的指令的下一条指令的地址

    执行引擎遇到任意方法返回的字节码指令，会有返回值传递给上层的方法调用者，简称正常完成出口

    在字节码指令中，返回指令包含ireturn（返回值是boolean、byte、char、short和int类型）、lreturn（long）、freturn（float）、dreturn（double）、areturn（引用数据类型），以及return指令供声明为void的方法、实例初始化方法、类的接口和初始化方法使用

    出现异常而退出的，返回地址通过异常表来确定，栈帧不保存这部分信息

    方法执行过程中出现异常（Exception），且该异常没有在方法内处理（即本方法的异常表中没有搜索到匹配的异常处理器），导致方法退出，称为异常完成出口

    正常完成出口有返回值，异常完成出口没有返回值

    无论哪种方式退出，在方法退出后都返回到该方法被调用的位置

  • 栈帧中的一些附加信息

    栈帧中还允许携带与Java虚拟机实现相关的附加信息，如对程序调试提供支持的信息

  • 面试题：方法中定义的局部变量是否线程安全？（具体情况具体分析）

    线程安全概念：如果只有一个线程才可以操作某数据，则是线程安全的

    如果有多个线程操作某数据，则此数据是共享数据，如果不考虑同步机制，会存在线程安全问题

本地方法栈

  • 本地方法（Native Method）

    本地方法就是Java方法调用非Java代码的接口（初衷是融合C/C++程序）

    定义本地方法时，不提供实现体，实现体由非Java语言在外面实现的

    标识符native可以与除abstract外的所有Java标识符连用

    使用本地方法的原因：与Java环境外交互、与操作系统交互、Sun的解释器是用C实现的

  • 本地方法栈（Native Method Stack）

    Java虚拟机栈用于管理Java方法的调用，本地方法栈用于管理本地方法的调用

    本地方法栈也是线程私有的

    内存溢出方面与虚拟机栈相同：超出分配的最大容量时抛出StackOverflowError异常；可动态扩容且尝试扩容时无法申请到足够内存、或创建新线程时没有足够内存创建本地方法栈时，抛出OutOfMemoryError异常

    当某个线程调用本地方法时，它不再受虚拟机限制（拥有和虚拟机同样的权限）

    不是所有JVM都支持本地方法，在HotSpot JVM中将本地方法栈和虚拟机栈合二为一

堆

一个JVM实例只存在一个堆内存

Java堆区在JVM启动时创建，其空间大小也就被确定了（堆内存的大小是可以事先设置的）

堆可以处于物理上不连续的内存空间中，但在逻辑上应该被视为连续的

所有线程共享堆，堆中可以划分线程私有的缓冲区（Thread Local Allocation Buffer，TLAB）

几乎所有的对象实例都在堆中分配内存，堆是执行垃圾回收的重点区域

在方法结束后，堆中的对象不会被马上移除，在垃圾回收时才会被移除

  • 堆内存细分

    Java7及之前，堆内存逻辑上分为三部分：新生区+养老区+（永久区，Permanent Space）

    Java8及之后，堆内存逻辑上分为：新生区+养老区+（元空间，Meta Space）

    新生区/新生代/年轻代：Young/New Generation Space

    养老区/老年区/老年代：Old/Tenure Generation Space

    永久区/永久代

  • 堆空间大小设置

    -Xms设置堆空间（新生代+老年代）的初始内存大小（等价于-XX:InitialHeapSize），-X是JVM运行参数，ms是memory start

    -Xmx设置堆空间（新生代+老年代）的最大内存大小（等价于-XX:MaxHeapSize）

    默认情况下，初始堆内存大小=物理电脑内存大小÷64，最大堆内存大小=物理电脑内存大小÷4

    堆区内存大小超过最大内存时，抛出OutOfMemoryError异常

    通常将-Xms和-Xmx设置成相同的值，在GC清理完堆区内存后不需要重新分隔计算堆区大小，提高性能

    查看设置的参数：方式一：命令行输入jps，jstat -gc 进程id

    方式二：idea中设置-XX:+PrintGCDetails

  • 新生代与老年代

    JVM中的Java对象分为两类，一种是生命周期较短的瞬时对象，这类对象的创建和消亡都非常迅速；另一类对象生命周期非常长，某些极端情况下能与JVM生命周期保持一致

    Java堆区可以分为新生代和老年代，新生代可以分为Eden空间、Survivor0空间和Survivor1空间（有时也叫做from区、to区）

    几乎所有Java对象都是在Eden区被创建

    绝大部分Java对象的销毁在新生代进行（IBM研究80%对象是“朝生夕死”的）

  • 配置新生代与老年代在堆中的占比：

    -XX:NewRatio：设置新生代与老年代的比例，默认值为2（新生代:老年代=1:2，新生代占堆的1/3）

    -XX:SurvivorRatio：设置新生代中Eden区与Survivor区的比例，默认值为8（伊甸园区:幸存者0区:幸存者1区=8:1:1，但实际比例为6:1:1）

    -XX:-UseAdaptiveSizePolicy：关闭自适应内存分配策略（暂时用不到）

    -Xmn：设置新生代空间大小（如果与比例设置相冲突，以此为准，一般不设置）

  • 对象分配过程

    1）新创建的对象先放在伊甸园区

    2）当伊甸园区空间被填满，再创建新对象时，JVM垃圾回收器将对伊甸园区进行垃圾回收（Minor GC），再将新对象放入伊甸园区

    3）将伊甸园区剩余对象移动到幸存者0区

    4）再次触发垃圾回收时，幸存者0区如果仍未被回收，则会放到幸存者1区

    5）之后每次触发垃圾回收且幸存者区未被回收的对象，会转移到另一个幸存者区

    6）当幸存者区的对象转移次数超过15次（默认）时，转移到老年代

    设置参数：-XX:MaxTenuringThreshold=<N>

    复制之后有交换，谁空谁是to区

    垃圾回收频繁发生在新生代，很少发生在老年代，几乎不发生在永久区/元空间

  • Minor GC、Major GC与Full GC

    JVM并非每次都对三个区域（新生代、老年代、方法区）一起回收，大部分回收发生在新生代

    HotSpot VM的GC按照回收区域分为：部分收集（Partial GC）和整堆收集（Full GC）

    部分收集：不是完整收集整个Java堆的垃圾收集，又分为：

    1）新生代收集（Minor GC/Young GC）：只对新生代（Eden、S0、S1）的垃圾收集

    2）老年代收集（Major GC/Old GC）：只对老年代的垃圾收集（目前只有CMS GC会有单独收集老年代的行为）

    3）混合收集（Mixed GC）：对整个新生代以及部分老年代的垃圾收集（目前只有G1 GC有这种行为）

    整堆收集：收集整个Java堆和方法区的垃圾收集（STW时间较长，开发中尽量避免）

  • 新生代GC（Minor GC）

    当Eden区满时，会触发Minor GC（Survivor满不会触发GC）

    Minor GC会引发STW，暂停其他用户线程，垃圾回收结束后用户线程恢复

  • 老年代GC（Major GC/Full GC）

    发生在老年代的GC，出现Major GC时，通常伴随至少一次的Minor GC（Parallel Scavenger收集器可以直接进行Major GC）

    当老年代空间不足时，先尝试触发Minor GC，若空间仍不足则触发Major GC

    Major GC速度一般比Minor GC慢10倍以上，STW时间更长

    若Major GC后空间仍不足，抛出OOM异常

  • Full GC触发情况：

    调用System.gc()时，系统建议执行Full GC（实际不一定执行）

    老年代/方法区空间不足时

    通过Minor GC后进入老年代的平均大小大于老年代的可用内存时

    由Eden或survivor space0（From Space）向survivor space1（To Space）复制时，如果对象大小大于To Space可用内存，则把该对象转存到老年代。如果老年代可用内存小于该对象大小，则触发Full GC

  • 内存分配策略（对象提升/Promotion规则）

    对象在Eden区被创建后经过Minor GC后存活，且能被Survivor区容纳，将被移动到Survivor区，对象年龄设为1. 以后每经历一次Minor GC且存活，年龄+1. 当年龄增加到一定程度（默认为15，不同JVM不同），会被晋升到老年代

    设置晋升老年代的年龄阈值：-XX:MaxTenuringThreshold

    创建出的对象如果大于Eden区，会被直接分配到老年代（尽量避免创建过多大对象）

    如果Survivor区中相同年龄的所有对象大小超过Survivor区空间的一半，年龄大于等于该年龄的对象直接进入老年代

    空间分配担保：-XX:HandlePromotionFailure

  • TLAB（Thread Local Allocation Buffer）

    TLAB是JVM为每个线程分配的私有缓存区域，包含在Eden区内

    由于堆是多线程共享的区域，多线程同时分配内存时可能存在线程安全问题，TLAB可以避免线程不安全

    TLAB能够提升内存分配的吞吐量，因此这种内存分配方式称为快速分配策略

    尽管不是所有对象都能在TLAB中成功分配内存，但JVM仍将TLAB作为内存分配的首选

    设置是否开启TLAB空间：-XX:UseTLAB

    TLAB默认只占有Eden区的1%，设置TLAB占用Eden百分比：-XX:TLABWasteTargetPercent

    对象在TLAB空间分配失败时，JVM会尝试使用加锁机制确保数据操作的原子性，直接在Eden区分配内存

堆空间参数设置：

命令 含义
-XX:+PrintFlagsInitial 查看所有参数的默认值
-XX:+PrintFlagsFinal 查看所有参数的最终值
-Xms 设置初始堆空间内存（默认为物理内存的1/64）
-Xmx 设置最大堆空间内存（默认为物理内存的1/4）
-Xmn 设置新生代大小（初始值及最大值）
-XX:NewRatio 设置老年代与新生代在堆结构中的比例
-XX:SurvivorRatio 设置新生代中Eden与S0/S1的比例
-XX:MaxTenuringThreshold 设置新生代垃圾最大年龄
-XX:+PrintGCDetails 输出详细GC日志
-XX:HandlePromotionFailure 是否设置空间分配担保

在发生Minor GC前，虚拟机会检查老年代最大可用的连续空间是否大于新生代所有对象的总空间：

1）如果大于，则此次Minor GC是安全的

2）如果小于，则虚拟机会查看-XX:HandlePromotionFailure是否设置允许担保失败

  1. 如果HandlePromotionFailure=true，会继续检查老年代最大可用连续空间是否大于历次晋升到老年代的对象的平均大小

    1.1. 如果大于，则尝试进行Minor GC，但这次Minor GC仍是有风险的

    1.2. 如果小于，则改为进行一次Full GC

  2. 如果HandlePromotionFailure=false，则进行一次Full GC

在JDK6 Update24之后，HandlePromotionFailure参数不再影响虚拟机的空间分配担保策略，只要老年代连续空间大于新生代对象总大小或大于历次晋升平均大小，则进行Minor GC，否则进行Full GC

  • 堆不是分配对象存储的唯一选择

    如果经过逃逸分析（Escape Analysis）后发现，一个对象并没有逃逸出方法，该对象就可能被优化成栈上分配

    开发中能使用局部变量的，就不要使用在方法外定义

  • 逃逸分析

    逃逸分析是一种可以有效减少Java程序中同步负载和内存堆分配压力的跨函数全局数据流分析算法

    通过逃逸分析，Hotspot编译器能够分析出某对象的引用的使用范围，从而决定是否将该对象分配到堆上

    逃逸分析的基本行为就是分析对象动态作用域：

    • 方法中定义的对象，如果只在方法内部使用，则认为没有发生逃逸
    • 方法中定义的对象，如果被外部方法引用，则认为发生逃逸

    JDK6u23之后，Hotspot默认开启逃逸分析

  • 使用逃逸分析，编译器可以对代码做出优化：

    1. 栈上分配。将堆分配转为栈分配（对象没有逃逸出方法），分配完后继续在调用栈内执行，最后线程结束，栈空间被回收，局部变量对象也被回收，不需要进行GC

      栈上分配常见场景：为成员变量赋值、方法返回值、实例引用传递

    2. 同步省略。如果一个对象只能从一个线程被访问到，对于该对象的操作可以不考虑同步。在动态编译同步块时，JIT编译器可以借助逃逸分析来判断同步块所使用的锁对象是否只能被一个线程访问。如果是，编译时会取消代码同步，提高并发行和性能。取消同步的过程称为同步省略，也叫锁消除

    3. 分离对象或标量替换。有的对象可能不需要作为一个连续的内存结构存在也可以被访问，该对象的部分（或全部）可以不存储在内存（堆），而是存储在CPU寄存器（栈）中

      标量（Scalar）是指一个无法再分解成更小数据的数据。Java中的原始数据类型就是标量

      可以被分解的数据称为聚合量（Aggregate），Java中的对象就是聚合量，可以被分解为其他聚合量和标量

      标量替换：经过逃逸分析，如果发现一个对象不会被外界访问，JIT优化后会把该对象拆解为若干个其中包含的成员变量来代替

  • 逃逸分析并不成熟

    根本原因在于无法确保经过逃逸分析的能量消耗小于原有方式的消耗（比如每个对象都是逃逸的），逃逸分析本身与要进行一系列复杂分析，是一个相对耗时的过程

    虽然逃逸分析技术并不成熟，但它是即时编译器优化技术中一个重要的手段

    Hotspot虚拟机中所有对象都是创建在堆上

方法区

方法区看作是一块独立于Java堆的内存空间。《Java虚拟机规范》中说明：“尽管方法区在逻辑上是属于堆的一部分，但一些简单的实现可能不会进行垃圾收集或者压缩。”Hotspot虚拟机的方法区还叫做Non-Heap（非堆）

方法区（Method Area）与堆一样，是各个线程共享的内存区域

方法区在JVM启动时创建，与堆一样，它的实际物理内存空间可以是不连续的

方法区的大小与堆一样，可以选择固定大小或可扩展

方法区的大小决定了系统可以保存类的数量，如果系统定义了太多类、加载大量第三方jar包、Tomcat部署工程过多等导致方法区溢出，会抛出异常：java.lang.OutOfMemoryError: Metaspace（新）或java.lang.OutOfMemoryError: PermGen space（旧）

关闭JVM会释放方法区内存

  • Hotspot中方法区的改进

    JDK7及以前习惯把方法区称为永久代，JDK8开始使用元空间取代永久代

    方法区和永久代并不等价，只是对于Hotspot而言是这样的

    元空间和永久代都是对方法区的实现，元空间不在虚拟机设置的内存中，而是使用本地内存（与JRockit、J9一样）

    永久代、元空间并不只是名字改变，内部结构也不同

    当年使用永久代，导致Java程序更容易OOM

  • 设置方法区大小

    JDK7及之前：-XX:PermSize设置永久代初始空间，默认为20.75M；-XX:MaxPermSize设置永久代最大空间，32位机器默认64M，64位机器默认82M

    JDK8及以后：使用-XX:MetaspaceSize和-XX:MaxMetaspaceSize代替原有参数。默认值依赖于平台，Windows下默认初始空间21M，最大空间值为-1，即没有限制（可以用尽所有系统内存）

    对于64位的服务端JVM，默认-XX:MetaspaceSize为21M，是初始高水位线。一旦触及该水位线，将触发Full GC并卸载没有用的类（即这些类对应的类加载器不再存活），高水位线将会重置（新的高水位线取决于GC后释放了多少元空间），如果释放空间不足，水位线适当提高；如果释放空间过多，水位线适当降低

    如果初始高水位线设置过低，高水位线将调整多次，发生多次Full GC

  • 如何解决OOM

    通过内存映像分析工具（如Eclipse Memory Analyzer）对dump出来的堆转储快照进行分析，重点是确认内存中的对象是否是必要的，也就是分清楚是内存泄漏（Memory Leak）还是内存溢出（Memory Overflow）

    如果是内存泄漏，可进一步通过工具查看泄露对象到GC Roots的引用连，定位到泄露代码的位置

    如果不存在内存泄漏，即对象必须存活，检查虚拟机的堆参数，看是否可以调大；代码上检查是否存在某些对象生命周期过长、持有状态时间过长的情况，减少程序内存消耗

  • 方法区存储内容

    方法区存储已被虚拟机加载的类型信息、常量、静态变量、即时编译器编译后的代码缓存等

    1. 类型信息

      对每个加载的类型（类class、接口interface、枚举enum、注解annotation），JVM在方法区存储以下信息

      1）该类型的完整有效名称（包名.类名）

      2）该类型直接父类的完整有效名称（interface和java.lang.Object没有父类）

      3）该类型的修饰符（public，abstract，final的某个子集）

      4）该类型直接接口的一个有序列表

    2. 域（Field）信息

      JVM必须在方法区保存所有域的相关信息以及域的声明顺序

      域的相关信息包括：域名城、域类型、域修饰符（public，private，protected，static，final，volatile，transient的某个子集）

    3. 方法（Method）信息

      JVM保存以下方法信息，同域信息一样包括声明顺序：

      方法名称、方法返回类型（或void）、方法参数的数量和类型（按顺序）、方法修饰符

      方法的字节码（bytecodes）、操作数栈、局部变量表及大小（abstract、native方法除外）

      异常表（abstract、native方法除外）：每个异常处理的开始位置、结束位置、代码处理在程序计数器中的偏移地址、被捕获的异常类的常量池索引

    4. non-final的类变量

      静态变量和类关联在一起随着类的加载而加载，成为类数据在逻辑上的一部分

      类变量被类的所有实例共享，即使没有类实例也可以访问

全局变量：static final

每个全局变量在编译时就被分配了

  • 运行时常量池与常量池

    方法区中包含运行时常量池，字节码文件中包含常量池

    常量池可以看做是一张表，虚拟机指令根据这张常量表找到要执行的类名、方法名、参数类型、字面量等类型

    常量池表（Constant Pool Table）是class文件的一部分，存储编译期生成的各种字面量与符号引用，这部分内容在类加载后存放到方法区的运行时常量池中

    JVM为每个已加载的类型（类或接口）都维护一个常量池，池中的数据项像数组一样，是通过索引访问的

    运行时常量池中包括在编译期就已经确定的数值字面量，也包括运行期解释后才能获得的方法或字段引用

    运行时常量池相对于常量池的重要特征是：具备动态性

    当创建类或接口的运行时常量池所需空间超过方法区能提供的最大值时，抛出OOM异常

  • 方法区演进细节

    JDK1.6及之前：有永久代（permanent generation），静态变量存放在永久代上

    JDK1.7：有永久代，字符串常量池、静态变量从永久代移除，保存在堆中

    JDK1.8及之后：无永久代，类型信息、字段、方法、常量保存在本地内存的元空间，字符串常量池、静态变量仍在堆中

  • 使用元空间替代永久代原因：

    1）永久代空间大小是难以确定的，如果动态加载类过多，容易产生永久代OOM。而元空间不在虚拟机中，使用本地内存

    2）对永久代进行调优是困难的

  • StringTable调整原因：

    JDK7将StringTable放到堆空间，因为永久代回收在Full GC时才会发生，StringTable回收效率不高。开发中会使用大量字符串，容易导致永久代内存不足

  • 方法区的垃圾收集

    《Java虚拟机规范》不要求对方法区实现垃圾收集（如JDK11时期的ZGC收集器不支持类卸载）

    类型卸载条件严苛，但方法区的收集有时又是必要的

    方法区的垃圾收集主要包括：常量池中废弃的常量和不再使用的类型

    方法区常量池中主要存放的两大类常量：字面量和符号引用。字面量比较接近Java语言中常量的概念，如文本字符串、被声明为final的常量值等；符号引用属于编译原理方面的概念，包括类和接口的全限定名、字段名称和描述符、方法名称和描述符

    Hotspot虚拟机只要常量池中的常量没有被其他地方引用，就可以被回收（与回收Java堆中的对象类似）

    回收类型需要满足三个条件：

    1）该类的所有实例都已经被回收，也就是堆中不存在该类及其任何派生子类的实例

    2）加载该类的类加载器已经被回收（很难达成）

    3）该类对应的java.lang.Class对象没有在任何地方被引用，无法通过反射访问该类的方法

    Java虚拟机允许对满足以上三个条件的类进行回收（不是必然被回收），Hotspot虚拟机提供了-XX:+TraceClassgc参数进行控制，还可以使用-verbose:class以及-XX:+TraceClass-Loading、-XX:+TraceClassUnLoading查看类加载和卸载信息

    在大量使用反射、动态代理、CGLib等字节码框架，动态生成JSP以及OSGi等频繁自定义类加载器的场景中，通常需要Java虚拟机具备类卸载能力

对象实例化

  • 创建对象的方式

    new（变形：Xxx的静态方法、XxxBuilder/XxxFactory的静态方法）

    Class的newInstance()：反射，只能调用无参构造器，权限必须是public

    Constructor的newInstance(Xxx)：反射，可以调用有参、无参构造器，权限没有要求

    使用clone()：不调用构造器，当前类需要实现Cloneable接口，实现clone()方法

    使用反序列化：从文件/网络中获取对象的二进制流

    第三方库Objenesis

  • 创建对象的步骤

    1. 判断对象对应的类是否加载、链接、初始化
    2. 为对象分配内存（如果内存规整：指针碰撞，如果内存不规整：虚拟机维护一个列表/空闲列表分配）
    3. 处理并发安全问题（采用CAS失败重试、区域加锁保证更新的原子性；每个线程预先分配一块TLAB）
    4. 初始化分配的空间（设置所有属性默认值，保证对象字段在不赋值时可以直接使用）
    5. 设置对象的对象头
    6. 执行init方法进行初始化

  • 对象内存布局

    1. 对象头（Header），包含两部分：

      1）运行时元数据（Mark Word）：哈希值（HashCode）、GC分代年龄、锁状态标志、线程持有的锁、偏向线程ID、偏向时间戳

      2）类型指针：指向类元数据InstanceKlass，确定该对象所属类型

      如果是数组，还需记录数组长度

    2. 实例数据（Instance Data）：是对象真正存储的有效信息，包括程序代码中定义的各类型字段（包括从父类继承的和本身拥有的字段）

      规则：相同宽度的字段分配在一起、父类定义的变量出现在子类前、如果CompactFields参数为true（默认为true），子类窄变量可能插入到父类变量的空隙

    3. 对齐填充（Padding）：非必须，仅起到占位符作用

直接内存

直接内存是在Java堆外、直接向系统申请的内存空间

直接内存不属于虚拟机运行时数据区，也不是《Java虚拟机规范》定义的内存区域

来源于NIO，通过存在堆中的DirectByteBuffer操作Native内存

通常，访问直接内存的速度优于堆，Java的NIO库允许Java程序使用直接内存作为数据缓冲区

  • 直接内存也可能出现OOM异常

    直接内存在堆外，大小不受堆空间限制，但堆和直接内存总和依然受限于操作系统最大内存
    直接内存分配回收成本高、不受JVM内存回收管理

    直接内存大小可以使用MaxDirectMemorySize设置（默认与堆空间最大值相同）

4. 执行引擎

执行引擎是Java虚拟机核心组成部分之一

物理机的执行引擎建立在处理器、缓存、指令集和操作系统层面上，而虚拟机的执行引擎由软件自行实现，可以不受物理条件制约指定指令集与执行引擎的体系结构，能够执行不被硬件直接支持的指令集格式

JVM的主要任务是负责装载字节码到其内部，执行引擎的任务是将字节码文件解释/编译为对应平台的本地机器指令

  • 解释器（Interpreter）与JIT编译器

    解释器：Java虚拟机启动时按照预定义的规范对字节码采用逐行解释的方式执行，将字节码转化为本地机器指令执行

    JIT（Just In Time Compiler）编译器：虚拟机将源代码直接编译成本地机器相关的机器语言

    Java是半编译半解释型语言：将解释执行与编译执行结合起来

  • 解释器

    为了满足Java语言跨平台特性，JVM采用实时解释器（而不是静态编译的方式直接生成本地机器指令）

    解释器相当于运行时“翻译者”，将字节码文件翻译为对应平台的本地机器指令执行

    当一条字节码指令被执行完后，再根据PC寄存器中记录的下一条指令进行解释操作

    JVM使用过两种解释执行器：字节码解释器（过去）、模板解释器（现在）

    1）字节码解释器在执行时通过纯软件代码模拟字节码的执行，效率低下

    2）模板解释器将每条字节码和一个模板函数相关联，由模板函数产生字节码对应的机器码，提高解释性能

    解释器在设计和实现上简单，目前基于解释器执行是低效的代名词

  • JIT编译器

    Hotspot VM采用解释器与即时编译器并存的结构，取长补短

    当程序启动后，解释器可以马上发挥作用；而编译器把代码编译成本地代码需要时间，但编译后执行效率更高

    对于服务端应用，启动时间并非关注重点：Java虚拟机启动时，解释器首先发挥作用，而不必等即时编译器全部编译完成；随着时间推移，编译器发挥作用，根据热点探测功能将有价值的字节码编译为本地机器指令，执行效率更高（在编译器进行激进优化不成立时，解释器是编译器的“逃生门”）

    机器在热机状态（启动运行一段时间后）可以承受的负载要大于冷机状态（刚启动）

  • Java中的“编译期”

    可能指前端编译器把.java文件转变为.class文件的过程（javac）

    也可能指虚拟机的后端运行编译器（JIT编译器）把字节码转变为机器码的过程（Hotspot的C1、C2编译器）

    还可能指使用静态提前编译器（AOT编译器）直接把.java文件编译成本地机器代码的过程

  • 热点代码及探测方式

    一个被多次调用的方法，或一个方法体内循环次数较多的循环体都可以被称为“热点代码”

    由于JIT编译器编译为本地机器指令发生在方法的执行过程中，因此也被称为栈上替换，或简称OSR（On Stack Replacement）编译

    Hotspot VM采用基于计数器的热点探测，为每个方法建立两个计数器：方法调用计数器（Invocation Counter）和回边计数器（Back Edge Counter）

  • 方法调用计数器

    用于统计方法被调用的次数，默认阈值在Client模式下为1500次，Server模式下为10000.超过阈值会触发JIT编译

    阈值可以通过-XX:CompileThreshold人为设定

    当一个方法被调用时，先检查该方法是否存在被JIT编译过的版本，如果存在，则优先使用编译后的本地代码执行。如果不存在，则方法调用计数器值+1，然后判断方法调用计数器与回边计数器只和是否超过方法调用计数器的阈值。如果超过阈值，会向即时编译器提交该方法的代码编译请求

  • 热度衰减

    方法调用计数器统计的并不是方法被调用的绝对次数，而是相对执行频率，即一段时间内方法被调用的次数。当超过一定时间限度，如果方法的调用次数仍不足以提交给即时编译器编译，该方法的调用计数器会减少一半。这个过程称为方法调用计数器热度的衰减（Counter Decay），这段时间称为方法统计的半衰周期（Counter Half Life Time）

    进行热度衰减是在虚拟机进行垃圾收集时顺便进行的，可以使用-XX:-UseCounterDecay关闭热度衰减（统计绝对次数），只要系统运行时间足够长，绝大部分代码都会编译成本地代码

    使用-XX:CounterHalfLifeTime设置半衰周期时间，单位是秒

  • 回边计数器

    用于统计方法中循环体代码执行的次数

    在字节码中遇到控制流向后跳转的指令称为“回边”（Back Edge）

  • HotSpot设置程序执行方式

    -Xint：完全采用解释器模式执行程序

    -Xcomp：完全采用即时编译器模式执行程序，如果即时编译出现问题，解释器会介入执行

    -Xmixed：采用解释器+即时编译器混合模式执行程序（默认情况）

  • HotSpot中JIT分类

    HotSpot内嵌有两个JIT编译器：Client Compiler和Server Compiler，简称为C1编译器和C2编译器

    -client：指定Java虚拟机运行在Client模式下，并使用C1编译器。C1编译器对字节码进行简单优化，耗时短

    -server：指定Java虚拟机运行在Server模式下，并使用C2编译器。C2编译器进行耗时长的优化及激进优化，但优化的代码执行效率更高

  • C1编译器和C2编译器的不同优化策略

    C1编译器优化：

    1）方法内联：将引用的函数代码编译到引用点处，减少栈帧的生成，减少参数传递及跳转过程

    2）去虚拟化：对唯一的实现类进行内联

    3）冗余消除：在运行期间把一些不会执行的代码折叠掉

    C2编译器优化主要在全局层面，基于逃逸分析：

    1）标量替换：用标量值代替聚合对象的属性值

    2）栈上分配：对于未逃逸的对象分配在栈而不是堆

    3）同步消除：清除同步操作，通常指synchronized
